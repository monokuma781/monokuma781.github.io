研究方向简介

机群文件系统

研究背景

1.         机群体系结构是构建高性能计算机的最通用的途径

机群技术从上个世纪90年代初提出到现在，经过十几年的发展，已经成为一种成熟的技术。机群体系结构具有高的性能价格比、良好的系统伸缩性、高可用性、易管理性和易使用性[1]，因此，它已成为构建高性能计算机的最通用的途径。 

2.       机群需要能被各节点共享的文件系统

在机群体系结构下，一台超级计算机是由若干节点构成。除不含键盘、鼠标和显示器之外，每个节点是一台完整的计算机，如PC机、工作站或SMP机器等。即每个节点都有自己的CPU、内存和磁盘，有自己的操作系统[1]。原来的文件系统是每个节点自己的文件系统，它只能为运行于本节点上的进程提供文件服务，而且它只能访问本节点自己的磁盘。我们称之为单机文件系统或本地文件系统（local file system）。从应用的角度看，它将机群系统中所有节点视为一个整体，即一台机器。一个并行应用可能需要同时在多个节点上运行，各个节点上的进程需要一个共享的文件系统。从机群系统的管理和其它应用的角度，机群系统各个节点也需要一个共享文件系统。

机群文件系统（Cluster file system）是在机群内各节点间共享的、单一系统映像的、高性能、易扩展、易管理、易使用的文件系统,它最基本的特征是：

(1)    各个节点共享的文件系统。即能从各个节点上对同一个文件系统进行访问。

(2)    具有全局的名字空间。即从各个节点上看到的文件系统结构是完全一样的，而且文件的名字与文件存储的位置无关。

(3)    对机群文件系统的访问采用OS系统调用接口和系统命令。即对机群文件系统的访问同对单机文件系统的访问采用完全一样的接口。使用机群文件系统与使用单机文件系统可以是没有差别的，单机文件系统上的应用无需编译就可直接运行于机群文件系统之上。 

3.       已有的文件系统不能很好地满足机群文件系统的需求

早期的分布式文件系统，如NFS、AFS等[2,3]，是基于网络上不同的计算机之间共享文件的思想发展起来的。将它们用作机群文件系统会带来一些问题（详见相关研究中的NFS和AFS部分）。

　

研究内容

机群文件系统需要解决下面几个关键问题：

1.         高性能。机群面向的是高性能科学计算应用、Internet应用、Video应用、数据库应用和DATA FARM等，这些应用无论是对聚合I/O带宽还是对元数据处理能力都有很高的要求。作为一个例子，美国DOE&DOD根据其下各个国家实验室的应用提出对机群文件系统的性能需求[4]，如表1、表2和表3 所示。表1给出的是并行I/O的聚合带宽需求。表2给出的是单个节点创建吞吐率的需求，对应的操作是对一个单一目录树包进行解包（untar）操作，目录树中所有文件的大小随机分布在100bytes―16Kbytes；平均每个目录下有5000个文件。表3给出的是多个节点同时在同一目录上创建文件的聚合吞吐率需求，其中N为节点总数，R为单节点的创建吞吐率。

聚合带宽是机群文件系统最关注的问题，在这方面已有大量的技术，包括：高速通讯网络的采用、高性能的通讯机制（如用户空间通讯、RDMA、OS旁路的通讯等），高性能的服务器和RAID，客户端缓存机制和预取机制等。

提高元数据处理的性能的研究则相对较少，除了采用客户端元数据缓存外，还没有特别有效的技术。

2.       可扩展。机群文件系统需要支持几个节点、几百个节点、甚至1024个节点的机群系统。如此大的规模，象NFS这样的单服务器网络文件系统是根本不可能支持的。需要多服务器文件系统，通过增加服务器数量来支持更多的节点。而且现在大量的应用都要求机群系统提供365×7×24的机时,因此迫切需要机群文件系统存储容量或机群文件系统服务器能够在线扩展，即不必停止机群文件系统的正常服务。另一方面，文件系统本身的规模也越来越大，这体现在以下三个方面：第一，文件系统大小和文件大小都越来越大，远远超过4GB，而是Terabytes级别，甚至是Petabytes级别，作为一个例子，表4是美国DOE&DOD根据其下各个国家实验室的应用提出对机群文件系统大小的需求[4]；第二，大目录，已经出现一个目录下106―107个文件的需求；第三，文件系统中文件数目非常大。

        多存储服务器技术较好地解决了带宽的扩展性。对于元数据处理，则有的是单一元数据服务器成为性能瓶颈，不可扩展；有的是无服务器结构，但这种无服务器结构通常都需要一个锁服务器，因此性能瓶颈转移为单一的锁服务器。从吞吐率来看，目前的机群文件系统还没有解决元数据处理的可扩展性。

3.       高可用或容错。由于机群系统是由许多部件（component）构成的，包括硬件部件（如节点，通讯卡、磁盘、通讯网络等）和软件子系统（节点操作系统、机群操作系统、机群文件系统等）。随着规模的扩大，这些部件出故障的机率也大大增加。机群文件系统必须具有容错性，即所谓的RSA，包括可靠性、可用性和可服务性（Reliability, Availability, Serviceability）。在个别部件或成员故障的情况下，机群文件系统能够做到：第一，保证文件系统不会崩溃，文件系统的结构不会被破坏，文件系统中的文件和数据不会丢失；第二，继续提供文件服务。

4.       性能评价。目前衡量机群文件系统的性能指标是聚合I/O带宽和聚合吞吐率。但仅有这两个指标还不能全面地刻画机群文件系统的性能。除了聚合I/O带宽和聚合吞吐率之外，需要新的性能评价的标准，例如用来评价机群文件系统的扩展性能、过载性能、服务质量等的评价标准。已有的文件系统benchmark并非是为机群文件系统设计的，一般都没有考虑并行文件访问模式、并发文件访问模式和共享访问模式，没有考虑扩展性能，不能很好地反映元数据处理的性能。此外，还需要有反映机群文件系统服务质量的Benchmark。

5.       可管理。需要机群文件系统提供系统管理、状态监控、性能调节（performance tuning）和内部统计信息收集等工具来辅助机群文件系统的管理和使用。

6.       对网格的支持。在网格环境里，机群文件系统需要提供一定的安全机制，需要支持不同管理域之间的文件共享。
