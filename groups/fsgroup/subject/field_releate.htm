                                   相关研究


    机群文件系统由分布式文件系统发展而来，它本质上仍是一种分布式文件系统，它与一般分布式文件系统的差异是由机群系统与普通分布式环境的差异带来的。例如，机群系统各个节点有高速网络互连，而普通分布式环境一般是性能较低的普通以太网互连；机群除了有共享文件的需求外，对机群文件系统的性能要求非常高，而普通分布式环境则主要解决共享文件的需求；机群作为一个整体，对高可用的要求非常高，远非普通分布式系统所能达到的；机群的节点位于同一个管理域中，节点间的文件共享是域内共享。

1. 早期的分布式文件系统
    早期的分布式文件系统是基于消息传递（Message-Based），为client-server模式。客户端需要经过文件服务器才能访问存储设备，维护UNIX文件共享语义的开销往往很大，出现了多种松散的文件共享语义，如会话语义、基于时间的一致性语义和事务处理语义等。NFS[2，3]和AFS[2，3]是这个时期的两个经典代表，直到现在还有非常广泛的应用。目前许多机群系统都以NFS作为其节点间共享的机群文件系统。

    NFS是SUN公司研制的，于1985年发布NFS2协议，1995年发布NFS3协议，目前NFS4协议正在讨论中，最新NFS4协议草案是2000年12月发布的。绝大数UNIX操作系统和一些非UNIX操作系统（如VMS和MS-DOS）都实现了NFS，NFS得到广泛地应用，成为事实上的工业标准。NFS采用client-server模型，单服务器结构，无状态协议使得节点崩溃容易恢复。NFS适用于中小规模的机群。作为机群中的共享文件系统NFS存在以下几个问题：第一，NFS是单文件服务器结构，而一个文件服务器所能支持的客户端数量是有限的，通常是不超过几十个节点。当机群系统的规模很大（几百个节点，甚至上千个节点）时，NFS无法支持如此多的节点。第二，NFS非UNIX语义。对于要求严格语义的应用，需要使用文件锁。而NFS下使用文件锁需要关闭客户端属性缓存，这使得NFS的性能大大降低而难于满足应用的要求。第三，由多个NFS服务器构建一个全局名字空间的文件系统，对某些应用环境（各个应用访问的文件分布在不同的NFS服务器上）来说可以解决一个文件服务器的瓶颈问题。但是当一个并行应用需要在大规模的节点上运行时，仍然出现运行在各个节点上的进程要访问同一个NFS服务器。再者这样的文件系统管理和维护起来相当困难，尤其当系统规模较大时。
    AFS是CMU与IBM联合研制的（项目开始于1982年，相关文章始见于1986年），发行了几个版本，AFS3.0是顶峰之作。其后，针对AFS的工作转移到Transarc公司，AFS演变为OSF的分布式计算环境（DCE）的分布式系统（DFS）组成部分。AFS扩展性好，能够扩展到几千个节点，提供一个统一的位置无关的名字空间。此外，AFS有较强的安全性（基于Kerberos）。但以AFS作为机群中的共享文件系统也存在一些问题[2]：第一，其性能不能令人满意，它使用本地文件系统来缓存最近被访问的文件块，但却需要一些附加的极为耗时的操作，结果，要访问一个AFS文件要比访问一个本地文件多花一倍的时间。第二，其容错性较差，由于它采用有状态模型，在服务器崩溃，网络失效或者其他一些像磁盘满等错误时，都可能产生意料不到的后果。第三，它不是UNIX文件共享语义，而是会话语义，即仅在每次文件打开或关闭时才进行缓存的一致性操作。这种语义比NFS基于时间的一致性更弱，不能满足大量的高性能科学计算应用的需求。

2. 机群上的并行文件系统
    由于对IO带宽有很高的需求，机群文件系统在发展中借鉴了并行文件系统和RAID的技术，出现了将一个文件的数据块分布于多个服务器上以提高一个I/O请求的并发度，如UCB的xFS，IBM的GPFS和Clemson大学的PVFS[16,17,18,19]。此外，有些机群文件系统还提供专门的并行I/O接口，如Clemson大学的PVFS。

    PVFS（Parallel Virtual File System）是Clemson大学开发的开放源码的并行文件系统，是一种基于消息传递的机群文件系统。它应用于Beowulf类型的LINUX机群系统上，主要面向大规模科学计算类型的应用。














    PVFS针对NFS单一服务器结构不可扩展进行改进，采用多IO服务器，单一元数据服务器结构。一个元数据服务器来维护一个全局的名字空间，而文件存储于多个IO服务器上（Striping）。文件块与IO服务器上子文件的映射由Striping策略决定，通过计算即可知道，因此提高了I/O性能。PVFS由若干客户节点，多个IO服务器和一个元数据服务器构成，如图2-6所示。PVFS支持64位文件，没有客户端缓存，提供三种接口：传统UNIX文件系统接口（通过OS系统调用和命令），用户级并行文件系统库（libpvfs.a）接口和MPI-IO接口。PVFS没有容错性，无论客户节点失效，还是IO服务器或元数据服务器失效都会使得PVFS不可用。PVFS没有实现POSIX文件锁，使得依赖文件锁的应用只得采用其它方式来同步。PVFS的可扩展性较差，所有节点必须在同一子网内。单一的元数据服务器是其性能瓶颈，当目录中文件个数为50,000个时，便非常慢（响应时间为几分钟）。稳定性有问题，当应用规模很大时（128个节点操作1000个2MB的文件），IO服务器和元数据服务器都会崩溃。
    目前存在研制中的PVFS V2，主要的改进在于提供文件复制和镜像以增强容错能力，采用多元数据服务器，支持多种网络协议（已支持TCP/IP和Myricom的GM），支持多种数据分布方式，可动态接入不同的数据分布机制等。

3. 基于共享存储的机群文件系统
   随着高速通讯网络技术的发展，出现了基于共享存储的机群文件系统，其特点是每个客户节点都可以直接访问存储设备，不需要文件服务器。有的没有任何服务器（全对称型），有的需要元数据服务器（非对称型）。这种结构的优势在于可扩展性好，增减存储设备非常容易，而且容易实现UNIX语义。由于一般采用高性能通讯网络（如HIPPI，Fibre Channel等）来连接客户节点和存储设备，基于共享存储的机群文件系统通常都有很高的性能。典型代表是IBM公司的GPFS[6,7]、SGI公司的CXFS[13,14,15]、Minnesota大学和Sistina公司的GFS[9,10,11,12]。
GPFS（General Parallel File System）IBM公司为SP机群系统开发的一种共享存储机群文件系统。适用于AIX机群环境和Linux机群环境。GPFS可以用于通用互连网络（如以太网或SP Switch）环境（如图2-1），GPFS节点通过VSD服务器访问磁盘；也可以用于存储局域网SAN环境（如图2-2），如FC-SAN或iSCSI，GPFS节点直接访问磁盘。






　

　


    GPFS是较成功的一个商业机群文件系统，其显著特点是性能高、扩展性好，高可用。在ASCI White上，492个节点，总共75TB的文件系统,GPFS可以获得接近7GB/sec的聚集带宽。GPFS通过增添硬件可以获得更高的性能。GPFS提供了故障恢复机制，其可靠在于两个方面：一方面是高可用性，即使文件系统中的一个节点死了，其它节点仍然能访问文件系统；另一方面是容错，即使文件系统中的磁盘失效了，文件数据也不会丢失。
GPFS的技术特色包括：客户端在内存缓存元数据和文件，用字节粒度的分布式锁实现了UNIX语义，文件数据在多个服务器上分条存储（data striping），利用Extendible Hashing技术提高大目录（100万个目录项）查找的性能，无元数据服务器，各个节点独立的元数据日志，基于日志的恢复技术，容错技术（节点失效、网络失效、存储设备失效）。

    GFS（Global File System）是Minnesota大学开发的基于SAN的共享存储的机群文件系统，如图2-4所示。后来Sistina公司将GFS产品化。GFS最初是在IRIX上开发的，后来移植到LINUX上，并开放源码。借助NBD，GFS可运用于LAN环境中，如图2-5所示。












    GFS与GPFS结构相似，但它是全对称的机群文件系统，没有服务器，因而没有性能瓶颈和单一故障点。GFS将文件数据缓存于节点的存储设备中，而不是缓存在节点的内存中。并通过设备锁来同步不同节点对文件的访问，保持UNIX文件共享语义。目前GFS实现了日志，节点失效可以快速恢复。GFS使用SCSI设备锁来进行同步，目前很少设备实现这种设备锁。在没有设备锁的情况下，GFS也是通过唯一的锁服务器来进行同步，因此，锁服务器是其性能的瓶颈。

    CXFS是SGI公司开发的基于FC-SAN的高性能机群文件系统，是一种非对称式的共享存储的机群文件系统，每个CXFS文件系统有唯一的元数据服务器，不同的CXFS文件系统的元数据服务器可以在不同的节点上。元数据服务器采用双备份来保证高可用。元数据和文件数据使用不同的网络，元数据使用LAN，文件数据使用SAN（如图2-3所示）。
CXFS客户端也将元数据和文件缓存在内存中，但与GPFS不同的是CXFS采用的是粗粒度的锁（文件粒度锁）来维护缓存一致性，实现UNIX语义。对于无节点间共享写的情况下，CXFS有较高的性能，但对于节点间的共享写效率较低。对元数据密集型操作，CXFS的性能不高，包括：打开和关闭文件，修改文件大小，创建和删除文件，目录查找。因此对大量小文件进行随机存取，反复地查找文件或目录信息这样的应用，CXFS的性能不高。由于采用文件粒度的锁机制进行同步，当多个节点对一个共享文件同时写或有读有写时，其I/O带宽不高。


    CXFS的高可用考虑三个方面：(1)文件系统节点是通过心跳消息监测，一旦发现一个节点失效，元数据服务则自动开始恢复步骤。元数据服务器检查XFS的日志，跳过所有失效节点尚未完成的操作，回溯到先前的某个一致点。(2)单一的元数据服务器采用在另一节点上运行备份元数据服务器来保证高可用。(3)网络和存储设备通过双网和高可用RAID来保证高可用。

    Lustre是Cluster File Systems, Inc.正在开发中的一个机群文件系统，如图2-7所示。Lustre分别用不同的机制来处理文件数据和文件系统元数据。它提出了一些较好的思想，包括：基于面向对象磁盘，元数据请求的批处理、面向网格的考虑（安全性：认证、授权和加密），支持通过网络来对Lustre进行管理和监控等。文件数据存储在面向对象的磁盘（Object-base Disk）上，用分布式的面向对象的存储服务器（OST，Object Storage Target）来负责真正的文件系统I/O和与存储设备的接口；用元数据服务器（MDS，Metadata Server）来存储和管理文件和文件系统的元数据。目前，Lustre是单一的元数据服务器，为了保证高可用性，在系统其它节点上运行一个复本（Replicated MDS）。Lustre打算将来实现多元数据服务器来提高元数据处理的性能和可扩展性。
Lustre在设计上考虑了对网格环境的支持，一方面它考虑了文件系统的安全性策略，包括认证、授权和文件内容加密三个方面；另一方面它考虑在管理与系统状态监控方面与网格的结合，其配置信息和状态信息采用XML格式，提供通过LDAP接口操纵配置信息，而且可以支持通过SNMP来获取和修改这些信息。


    目前，Lustre只有0.5.16版，只实现了较少的功能，而客户端缓存一致性、多元数据服务器、安全性策略等都还没有实现。

4. 小结
    综上所述，我们不难看出，机群文件系统发展趋势有四个：第一，与并行文件系统结合，除了实现通用文件系统接口之外，有些机群文件系统还实现专门的并行文件系统接口，并对之进行优化，使得通过并行文件系统接口可以得到更高的性能，以支持高性能科学计算应用。第二，借鉴并行文件系统和RAID的数据striping的思想，机群文件系统让一个文件的数据分布在多个存储服务器上（data striping）。第三，存储服务器的专用化，即在机群中有专门的存储节点，提供机群文件系统的存储。第四，近年来，随着网络存储尤其是存储局域网（SAN）的发展，出现建筑在共享网络存储之上的机群文件系统的趋势。
